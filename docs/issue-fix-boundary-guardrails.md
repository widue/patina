# 稳定期问题修复边界守则

## 1. 文档定位

本文定义 `Patina` 进入稳定期后的默认问题修复方式。

它不是一次性执行单，也不是长期架构文档的替代品，而是用来回答下面 4 个问题：

- 当前问题属于单点小修、边界判断，还是结构性执行单
- 这次修复的真实 owner 是谁
- 哪些修改是允许的，哪些会把边界重新弄脏
- 修完之后至少要验证到什么程度

如果某次局部实现习惯与 [`architecture.md`](./architecture.md) 冲突，以 `architecture.md` 为准。

---

## 2. 与其他长期文档的关系

- [`architecture.md`](./architecture.md) 定义长期结构、owner、边界与禁止事项；本文定义稳定期遇到问题时怎样按这些规则做判断。
- [`product-principles-and-scope.md`](./product-principles-and-scope.md) 解决“这个方向值不值得做”；本文解决“这次问题该怎么修才不把边界弄脏”。
- [`quiet-pro-component-guidelines.md`](./quiet-pro-component-guidelines.md) 约束 UI 呈现方式；本文约束问题修复时的落点与扩散范围。
- [`roadmap-and-prioritization.md`](./roadmap-and-prioritization.md) 约束优先级；本文约束具体修复方式。
- [`versioning-and-release-policy.md`](./versioning-and-release-policy.md) 约束发布前的版本与验证；本文约束日常修复与收口方式。

## 3. 稳定期默认原则

### 3.1 先判断 owner，再判断怎么修

稳定期最常见的风险不是“大重构做坏”，而是“小修复慢慢把边界重新打散”。

因此默认顺序是：

1. 先判断这个问题属于哪个 owner
2. 再判断最小修复是否还能留在 owner 边界内
3. 只有确认边界后，才进入实现

### 3.2 小问题也不能按“哪里顺手改哪里”处理

以下做法默认都算越界信号：

- 因为图快，把逻辑补进当前最容易改的壳层文件
- 为了修一个点，顺手新增 page-local helper、临时 facade 或万能 util
- 让页面重新直接碰基础设施、SQLite 或 runtime 细节
- 让 `app/*`、`commands/*`、`lib.rs`、`shared/*` 承接新的厚逻辑

### 3.3 兼容壳可以存在，但必须变薄

稳定期允许少量兼容壳、转发层、历史适配层继续存在，但它们只能：

- 承担兼容旧调用或平滑过渡
- 做转发、组合、类型兼容
- 明确指向真实 owner

它们默认不能：

- 承接新的 feature 私有规则
- 承接新的平台实现
- 演变成新的长期落点

注意不要把所有带有 `legacy`、`migration` 或旧版本语义的代码都视为可删除兼容壳。旧产品身份、旧路径、旧 localStorage key 这类迁移窗口兼容可以按退出条件清理；但当前数据库 schema migration、repair 和已安装版本直升保护属于升级可信链路，默认不是小修模式里的“兼容壳清理”。如果要动这类代码，应先升级为边界判断或执行单。

## 4. 三种处理模式

## 4.1 小修执行模式

适用条件：

- owner 清晰
- 最小修复能留在同一层或同一 feature 内
- 不需要新增跨层抽象
- 不需要迁移职责

执行要求：

- 只在真实 owner 边界内做最小修复
- 不顺手新增跨 feature 的共享能力
- 不把逻辑移到更方便修改但 owner 不正确的位置
- 不把兼容壳写成新主路径

典型例子：

- 某个 feature 内部格式化、映射、状态分支的小回归
- 已有 gateway 内的参数或契约修正
- Quiet Pro 现有组件体系内的样式/状态修复

## 4.2 边界判断模式

适用条件：

- 问题不大，但落点不完全清晰
- 有两个以上看起来都“能改”的位置
- 一旦写错层，后续很容易继续扩散

执行要求：

- 先明确推荐 owner
- 先明确允许修改的层
- 先明确禁止扩散的层
- 边界判断完成后，再降级为小修执行或升级为执行单

典型例子：

- 同时涉及 `feature`、`shared`、`platform` 的行为修复
- 同时涉及前端读模型与 Rust runtime 行为的联动问题
- 看起来像 UI bug，但本质上牵涉数据边界或平台契约

## 4.3 执行单模式

适用条件：

- 修复必然跨层
- 修复过程需要迁移职责
- 当前暴露的是结构性债务，不是单点 bug
- 如果不先锁范围，实施过程极易一边修 bug 一边隐性重构

执行要求：

- 先写执行单，再进入实现
- 先锁目标、范围、非目标、阶段与验收门槛
- 明确真实 owner、兼容策略和退出条件

典型例子：

- 某条高频路径反复因为边界错误回归
- 需要把厚逻辑从壳层、页面层、命令层迁回真实 owner
- 需要收口运行时、SQLite、IPC 之间的职责

---

## 5. 默认判断顺序

以后遇到一个问题，默认按下面顺序判断：

1. 这是单点行为回归，还是结构性问题的表象。
2. 真实 owner 是哪个 feature、哪个 shared 能力、哪个 platform gateway，或哪个 Rust 层。
3. 最小修复能不能留在这个 owner 内部完成。
4. 如果不能，是进入边界判断，还是直接进入执行单。

可以把它压缩成一句话：

先问“这是谁的问题”，再问“怎么修”。

---

## 6. 必须先停下来的信号

出现以下任一情况，就不应继续按“小修执行模式”推进：

- 需要新增 `shared/*` 才能落地，但又说不清这项能力为什么是稳定共享能力
- 需要让页面、壳层或 feature 直接访问 SQLite、runtime 或平台细节
- 需要让 `app/*`、`commands/*`、`lib.rs` 承接新的厚逻辑
- 需要新增跨层端口、改变端口所有权，或让 `app/*` 新增跨 owner 组合关系
- 需要把职责从 engine/domain 迁到 data/platform，或反向迁移
- 需要把逻辑塞回已退出的根层 `src/lib/`、`src/types/`，或制造等价的新临时桶
- 需要新增兼容壳，但说不清真实 owner 与未来退出条件
- 修复看起来像 bug，但本质上已经是职责错放或边界缺失
- 连实施者自己也说不清“这段逻辑到底应该放哪”

一旦出现这些信号，默认先做边界判断或执行单，不继续直接写实现。

---

## 7. 与 `architecture.md` 对齐的高风险区

稳定期要重点盯防这些高吸力区域重新长厚：

- 前端 `app/*`
- 前端 `shared/*`
- 前端 `platform/*`
- Rust `lib.rs`
- Rust `commands/*`

这些层默认可以编排、转发、暴露边界，但不应重新承接厚业务逻辑、临时例外或私有规则。

---

## 8. 默认验证门槛

验证应与改动风险匹配，但下列情况必须至少满足 [`architecture.md`](./architecture.md) 中的最低门槛：

- 触及 tracking 主链、runtime 行为、session 边界
- 触及 history/dashboard 读模型
- 触及 IPC 契约、SQLite 读写边界、更新/备份/设置主路径
- 触及高风险层收口或兼容壳调整

默认最低门槛：

- `npm run check`

它会串联当前前端边界检查、关键测试、构建与 bundle 预算。局部修复可以先运行命中的专项测试，但交付前应按风险回到默认门槛。

如果涉及架构边界、Rust 核心路径或发布前复核，应运行：

- `npm run check:full`

如果涉及 IPC 注册、capability、plugin SQL 或真实桌面 runtime，还应在 Windows 运行 `npm run test:tauri-runtime-smoke`；如果涉及性能敏感的 read model、SQLite 查询或导航路径，还应运行 `npm run perf:stable`。新增跨层端口或 composition 关系时，必须同时通过前端/Rust 边界检查器及其 self-test，不能用临时 allowlist 代替 owner 判断。

如果涉及版本号、changelog、updater artifact 或发布脚本，还应补充当前发布校验脚本。

---

## 9. 验收时真正要看的事

稳定期验收不只看“功能有没有修好”，还要看下面 5 件事：

1. 代码有没有留在真实 owner 层里。
2. 有没有为了修一个点新增新的跨层抽象、兼容壳或公共垃圾桶。
3. 有没有让页面、壳层、命令层重新直接碰基础设施。
4. 有没有让高风险层变厚。
5. 有没有把本来该升级处理的问题伪装成了小修。

如果功能修好了，但这些问题明显出现，就不应算合格的稳定期修复。

---

## 10. 给 Codex 与后续协作者的默认约束

默认执行约束如下：

- 先按本文判断模式，不直接按手感开改
- 新逻辑优先落到真实 owner，不落到临时中间层
- 兼容壳只能变薄，不能继续长成主路径
- 需要跨层迁移时，先升级为执行单
- 任何一次修复都不应以“以后再整理”为理由制造新的长期例外

只有这样，稳定期才是真的在“持续收口”，而不是一边修一边回流。
