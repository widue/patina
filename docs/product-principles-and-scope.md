# 产品原则与范围

## 1. 文档定位

本文定义 `Patina` 的长期产品原则、目标范围与非目标范围。

它不是某一轮需求整理，也不是某个版本的功能清单，而是以后判断下面这些问题时的长期基线：

- 这个方向值不值得做
- 这个能力应做到什么程度
- 这个提案是否已经越过当前产品边界
- 这个想法现在该进入主线，还是应该延后

如果某次功能讨论与本文冲突，默认先回到产品原则做判断，而不是按“有人提了就做”处理。

## 3. 产品一句话定义

`Patina` 是一个面向个人桌面工作的、本地优先、自动记录、可回看且可控制的时间追踪工具。

这句话里的关键词都很重要：

- `个人`：当前优先服务单个使用者，而不是团队协作场景。
- `桌面工作`：优先围绕 Windows 桌面上的真实工作流，而不是移动端习惯。
- `本地优先`：默认假设本地数据即成立，用户拥有保存、备份、恢复与清理控制权。
- `自动记录`：核心价值来自自动采样与后续校正，而不是要求用户坚持手工打点。
- `可回看`：不只是采集数据，还要能让用户看懂、校正和使用这些记录。
- `可控制`：用户必须能理解系统在做什么，并对关键数据行为保有控制。

---

## 4. 当前核心用户

当前长期优先服务的用户是：

- 在 Windows 桌面上长时间工作的用户
- 希望低打扰地了解自己时间分布的用户
- 不想依赖账号、订阅或云端服务才能使用核心能力的用户
- 需要本地备份、恢复和清理控制权的用户

当前不优先服务的用户包括：

- 需要多人协作、团队报工、组织权限控制的用户
- 以移动端为主要使用场景的用户
- 把产品当作游戏化激励工具、社交番茄钟工具或绩效系统的用户
- 需要完整 SaaS 报表、审批流或企业计费体系的用户

---

## 5. 产品要解决的核心问题

## 5.1 自动记录问题

用户不想依赖手工开始/停止计时器来保证记录完整性。

因此产品应优先提供：

- 自动识别当前前台应用
- 稳定的会话切分
- 对 `AFK`、锁屏、睡眠等边界的正确处理

## 5.2 可相信的问题

时间追踪工具一旦让用户怀疑“它是不是记错了”，就会迅速失去长期价值。

因此产品应优先提供：

- 可解释的统计逻辑
- 尽量少漏记、少串记的边界处理
- 本地优先的数据存储与恢复能力
- 清晰的用户控制权

## 5.3 可读可管的问题

原始采样本身并不等于可用产品，用户还需要看懂并管理这些记录。

因此产品应优先提供：

- 可读的 `Dashboard`、`History` 与 `Data`
- 可管理的应用分类、重命名、配色与排除
- 可主动使用但不喧宾夺主的 `Tools / 工具`
- 明确的标题记录与行为控制

## 5.4 长期陪伴的问题

这个产品不是短期炫技工具，而是长期陪伴型桌面工具。

因此产品应优先提供：

- 安静稳定的体验
- 可解释的数据边界
- 长时间使用不累的界面
- 可渐进演进的结构与版本体系

---

## 6. 长期产品原则

### 6.1 可信优先于功能堆叠

如果一个新能力会削弱用户对记录准确性、数据边界或产品稳定性的信任，它就不应优先于核心追踪正确性。

### 6.2 自动优先于手工流程

产品的主要价值来自“自动记录 + 后续校正”，而不是让用户维护一套复杂的手工记账系统。

允许有手动校正能力，但不应把主路径变成手工流程。

### 6.3 本地控制优先于外部依赖

长期默认假设：

- 本地存储足以成立
- 备份/恢复是核心能力
- 不把账号体系、在线依赖、云端配置当作默认前提

### 6.4 清晰优先于炫技

无论是产品能力还是界面呈现，都应优先帮助用户理解自己的时间，而不是增加表演性体验。

### 6.5 架构契合是准入条件，不是事后优化

一个提案如果只有在打破 [`architecture.md`](./architecture.md) 中的 owner 规则、通道边界或高风险层约束时才能实现，默认不应进入当前主线。

### 6.6 渐进优先于铺得太大

个人开源产品最容易失败的方式之一，是过早把目标扩成“大而全平台”。

因此长期原则是：

1. 先把核心路径做可信。
2. 再把高频体验做顺。
3. 最后再考虑较大的横向扩张。

---

## 7. 当前长期范围

当前属于长期主方向的能力包括：

### 7.1 可信的自动追踪内核

- 前台应用识别
- 会话切分
- `AFK` / 锁屏 / 睡眠边界处理
- 崩溃恢复与异常封口
- 对视频、会议、课程、直播等低交互但真实参与场景，优先采用“信号驱动 + 状态机 + 可解释诊断”的持续参与识别，而不是应用白名单直接放宽

### 7.2 可读的时间回看能力

- `Dashboard`
- `History`
- `Data`
- 聚合统计、趋势、热力图与时间线解释一致性

### 7.3 可管理的应用语义层

- 应用分类
- 重命名
- 配色
- 排除统计：应用或网页域名处于排除状态时停止新增记录，并从常规统计中隐藏既有历史；恢复后既有历史重新显示，排除期间不补记
- 标题记录开关

### 7.4 本地数据控制

- 本地数据库
- 导出备份
- 恢复备份
- 清理历史记录

### 7.5 安静专业的桌面体验

- Quiet Pro 基线下的核心页面体验
- 最小但稳定的设置能力
- 与 Windows 桌面行为一致的产品感受

### 7.6 轻量主动工具页边界

允许在长期信息架构中保留 `Tools / 工具` 页面，用于承载用户主动打开、操作、观察状态的轻量本地桌面工具，例如提醒、计时器或番茄钟。

这类能力的默认边界是：

- `Tools / 工具` 放主动操作型能力
- `Settings / 设置` 放偏好配置型能力
- 工具到期提醒属于用户主动启动后的默认反馈，不在 `Settings / 设置` 暴露开关
- 第一版不把工具默认时长、状态入口或通知行为做成全局隐藏设置
- 工具页不替代自动追踪主路径
- 工具页不扩张成任务管理、项目管理、日程系统、团队协作或游戏化生产力平台

---

## 8. 当前明确非目标

以下方向当前不属于长期主方向，除非产品阶段发生明确变化：

### 8.1 团队协作与 SaaS 化

- 团队工作区
- 多人共享报表
- 管理员权限体系
- 在线组织与成员管理
- 云端订阅计费

### 8.2 重型生产力平台化

- 项目管理平台
- 任务系统
- 完整 `OKR / KPI / 报工` 体系
- 企业级运营后台

### 8.3 娱乐化效率产品路线

- 强游戏化激励
- 连续签到、成就墙、社交排行榜
- 用刺激感替代可相信度的交互设计

### 8.4 过早的多平台扩张

- 手机优先体验
- Web SaaS 首发
- macOS / Linux 的同步铺开

### 8.5 与核心价值弱相关的花式洞察

- 复杂 AI 总结
- 大量装饰性可视化
- 与高频核心需求无强关系的“高级分析实验”

---

## 9. 新功能准入标准

一个新提案要进入当前长期主线，至少应同时满足大部分条件：

- 它能增强自动记录、可信度、可回看性或本地控制力中的至少一项。
- 它服务的是当前核心用户，而不是假想中的另一类产品用户。
- 它不会明显削弱现有主路径的简单性与可信感。
- 它不要求仓库立即承担完全不同的产品形态。
- 它的长期维护成本与产品收益大体匹配。
- 它可以在 [`architecture.md`](./architecture.md) 的结构边界内稳定落地。

如果一个提案更像是：

- 看起来很酷
- 适合截图展示
- 只服务低频边缘场景
- 会把产品带向另一类工具

那它默认不应进入当前主方向。

---

## 10. 延后或否决的默认口径

以下情况通常应优先延后，而不是立刻实现：

- 与当前核心价值关系弱
- 长期维护成本过高
- 会显著拉高 UI 或架构复杂度
- 主要服务未来假想规模，而不是当前真实使用
- 会把产品从“个人桌面工具”拉向“平台型产品”

以下情况通常应直接考虑否决：

- 会破坏本地优先立场
- 会让数据边界更难解释
- 会把产品重心转向团队 SaaS、游戏化或营销型工具
- 会要求大规模新表面，但缺少稳定用户价值支撑

---

## 11. 当前阶段最重要的产品判断

当前阶段最重要的不是“功能面更大”，而是：

- 让追踪结果更可信
- 让数据控制更稳
- 让核心页面更清晰
- 让产品边界更自洽
- 让用户更愿意把它当成长期桌面工具

所以当两个方向竞争资源时，默认优先级应是：

1. 可相信的记录与数据安全
2. 高频核心页面的可读与可管
3. 结构稳定与长期可维护
4. 新扩展能力

---

## 12. 什么时候更新本文

只有在以下情况发生时，才应更新本文：

- 核心用户发生变化
- 长期主方向发生变化
- 某类原本的非目标已被正式纳入主线
- 产品阶段发生变化，例如从当前 `1.x` 稳定期进入新的产品阶段

如果只是某轮执行、一次修复或短期实验，不应频繁改动本文。

---

## 13. 给 Codex 与后续协作者的默认约束

当仓库内出现新想法、新 feature 或产品扩张提案时，默认先用本文判断它属于哪一类：

- 强化核心价值
- 改善现有主路径
- 边缘增强
- 明显越过当前产品边界

默认行为应是：

- 优先加深当前核心产品，而不是横向扩张成另一类工具
- 优先补可信度、清晰度和控制力，不优先补表演性能力
- 遇到明显跨产品方向的提案时，先停下来确认，而不是直接实现
