# Quiet Pro 组件规范

## 1. 文档定位

本文定义本项目长期使用的 `Quiet Pro` UI 规范。

它不是某一轮美化任务的执行单，也不是某个页面的局部设计说明，而是以后新增组件、扩展页面和修复界面时都应默认遵循的长期母规则。

---

## 2. 与其他长期文档的关系

- [`architecture.md`](./architecture.md) 约束代码结构与 owner；本文约束视觉与组件体系应该怎样长。
- [`product-principles-and-scope.md`](./product-principles-and-scope.md) 定义产品边界；本文定义这些产品能力应该以什么气质呈现。
- [`issue-fix-boundary-guardrails.md`](./issue-fix-boundary-guardrails.md) 约束 UI 问题怎样修才不越界；本文约束允许形成什么样的长期视觉规则。

如果某个局部方案与本文冲突，默认先确认是否真的需要新的长期设计方向，而不是让单页方案覆盖母规则。

---

## 3. Quiet Pro 一句话定义

`Quiet Pro = 安静、专业、克制、可长时间使用的桌面效率工具界面。`

它不追求“第一眼惊艳”，而追求：

- 信息清晰
- 层级稳定
- 长时间使用不累
- 不像模板化 AI 设计
- 后续容易继续扩展

---

## 4. 当前产品语境

本项目是一个：

- `Tauri` 桌面应用
- 时间追踪与工作陪伴型工具
- 以 `Dashboard / History / Data / App Mapping / Tools / Settings / About` 为核心工作面的产品
- 以更新状态、备份恢复、桌面挂件等能力作为低噪音支持面的产品

因此 UI 默认应更接近：

- 桌面工具
- 工作台
- 稳定系统
- 长时间持续使用的软件

而不是：

- 营销页
- 创意展示页
- 娱乐型效率产品
- 视觉实验场

---

## 5. 核心设计原则

### 5.1 信息优先于装饰

视觉的第一职责是帮助用户理解信息和完成操作，而不是制造“设计感”。

优先使用这些手段建立质量感：

- 字重
- 间距
- 对齐
- 留白
- 信息分组
- 语义清晰的状态层级

### 5.2 中性系统优先于单页特效

界面整体应保持中性、专业、耐用。

即使页面有重点区域，也应通过系统化样式表达，而不是靠单页独有的强效果抢眼。

### 5.3 产品气质重于内容皮肤

本项目更像桌面效率工具，不是品牌展示页。

因此组件应优先体现：

- 可靠
- 清晰
- 稳定
- 易扫读

### 5.4 主题变化必须建立在稳定骨架之上

允许继续演进主题，但主题变化应建立在稳定骨架之上：

- token
- 语义颜色
- 圆角尺度
- 边框/阴影尺度
- 密度规则

不应通过页面内部大量硬编码样式来“换皮”。

### 5.5 好看必须可复用

Quiet Pro 不反对设计感，但要求这种设计感能被系统化复用。

如果某个效果：

- 只能在一个页面成立
- 依赖页面独有背景
- 依赖大量硬编码值
- 复制到其他页面就违和

那它通常不属于 Quiet Pro 认可的长期做法。

---

## 6. 明确允许与明确禁止

## 6.1 允许的整体气质

- 冷静
- 克制
- 专业
- 中性
- 有桌面工具感

## 6.2 明确禁止

- 玻璃拟态
- 重模糊
- 霓虹发光
- 大面积炫目渐变背景
- 过度圆角
- 为了“高级感”堆大阴影
- 单页面专属花哨特效
- 明显的 AI 模板感

## 6.3 对“高级感”的正确理解

Quiet Pro 的高级感来自：

- 排版干净
- 层级明确
- 控件一致
- 颜色克制
- 节奏稳定
- 比例准确

而不是来自：

- 大面积装饰
- 发光
- 强品牌色占满页面
- 到处悬浮与漂浮
- 为显得贵而故意复杂

## 7. Token 与样式规则

### 7.1 一切优先走语义 token

新增视觉角色时，优先：

1. 复用现有 token
2. 如果现有 token 语义不够，再新增语义 token

不要直接在组件里硬编码：

- 新颜色
- 新圆角
- 新阴影
- 新边框风格
- 新动效尺度

### 7.2 Token 必须描述语义，不描述一次性外观

优先形成的是下面这类语义角色：

- `panel`
- `control`
- `chip`
- `status`
- `track`
- `muted text`
- `danger`

而不是“这个页面看起来大概需要一个什么值”。

### 7.3 分类色可以丰富，但外围 chrome 必须克制

分类色、图表色、状态点颜色允许相对丰富，但只服务业务识别。

周围界面 chrome 仍应保持 Quiet Pro 的中性骨架，不要让彩色元素反过来主导整页。

外部品牌色属于例外，只能在确实表达外部品牌、平台或第三方入口时使用，并且应收敛到最小元素，例如图标、品牌词或一小段文本。不要为了品牌色把整枚按钮、整张卡片或整块页面 chrome 改成强色。

### 7.4 尺度应可预测

长期上，控件应尽量收敛在有限的档位内：

- 圆角档位
- padding 档位
- 字级层级
- 边框与阴影强度

Quiet Pro 不鼓励“这个页面再发明一个更顺眼的新档位”。

### 7.5 样式事实要有清楚 owner

`src/App.css` 长期只作为样式入口，汇总导入 `src/styles/` 下的 CSS-only 文件。当前样式 owner 口径为：

- `src/styles/tokens.css`：Quiet Pro token、主题 scheme 与语义变量
- `src/styles/quiet-pro.css`：可复用组件原型
- `src/styles/app-shell.css`：应用壳层、通用页面与响应式规则
- `src/styles/features/*`：对应 feature 的局部样式

主题 swatch 与可选主题列表这类结构化主题数据不应留在页面组件内部，优先放在稳定的 shared settings/theme owner 中，再由页面组件消费。

## 8. 页面骨架与原型

## 8.1 壳层

应用壳层应保持：

- 稳定导航
- 清晰页头
- 主内容区边界明确
- 页面切换不过分戏剧化

壳层服务的是长期使用秩序，不承担视觉表演。

## 8.2 面板

面板是本项目最常见的容器原型。

它应体现：

- 清晰分组
- 克制边界
- 足够留白
- 不依赖重阴影建立层次

## 8.3 页头

页头应承担：

- 页面身份识别
- 简短上下文提示
- 必要的 controls、status 与 actions

页头不应承担：

- 大片品牌展示
- hero 式舞台
- 页面私有的强视觉实验

## 8.4 列表与工作台

列表、映射表单、设置区块应更像“工作台”，而不是“卡片墙”。

优先保证：

- 行级可扫读
- 信息密度平衡
- 操作入口稳定
- 状态对比明确

## 9. 组件规则

### 9.1 组件原型优先复用

新组件优先归入已有原型之一：

- `panel`
- `control`
- `chip`
- `status`

如果无法归类，再考虑新增原型，而且必须非常谨慎。

### 9.2 状态必须完整

新组件至少应定义：

- `default`
- `hover`
- `active`
- `focus`
- `disabled`

需要时再补：

- `loading`
- `empty`
- `selected`
- `error`

不能只做静态样子，不定义交互状态。

### 9.3 图标只做辅助识别

图标的职责是辅助识别，而不是成为视觉主角。

因此：

- 图标尺寸要克制
- 图标颜色不抢主信息
- 图标背景只服务识别，不应变成装饰噪音
- 同一组同层级 action 里的图标默认保持同一视觉强度；只有明确语义需要时，才允许某个图标单独使用 `danger / warning / success` 等语义色

### 9.4 按钮与输入控件应利落、清晰、可预期

默认不要引入：

- 过分圆润
- 拟物感
- 强阴影营造的“高级感”

同一组 action 按钮默认应使用统一的按钮原型、边框、背景、尺寸和 hover 反馈。不要因为某个按钮是支持、反馈、外链或品牌入口，就默认给它单独的边框、背景、阴影或更强 hover；如果需要强调，只优先在图标或小段文本上表达。

### 9.5 状态类组件必须有语义

`badge`、`chip`、`status` 之类组件只应用于：

- 当前状态
- 风险提示
- 筛选条件
- 轻量标签

不应用于：

- 大块装饰
- 替代标题
- 没有语义必要的“彩色点缀”

## 10. 排版、密度与数据展示

### 10.1 排版质量优先

Quiet Pro 的质量感很大程度来自排版。

优先保证：

- 标题层级清楚
- 副文案简短
- 数字与时间信息易扫读
- 长段说明尽量少

### 10.2 密度要服务效率，不服务压迫感

本项目是效率工具，不是海报。

默认应保持：

- 比营销页更高的信息密度
- 比传统后台更克制的噪音
- 在紧凑和可读之间偏专业的平衡

### 10.3 图表与分类色只服务信息理解

图表可以丰富，但图表周围的 UI 不应跟着一起变花。

如果分类色较多，应主动压低：

- 轨道色
- 面板边框
- 背景层次

---

## 11. 动效规则

Quiet Pro 的动效应当：

- 短
- 轻
- 有反馈意义
- 不打断使用节奏

可以有：

- 页面切换的轻过渡
- 悬停反馈
- `toast / modal` 的短入场

不应有：

- 大幅位移动画
- 浮夸弹跳
- 为了显得灵动而增加的表演性动效

去掉动效后，组件和页面仍应成立。

---

## 12. 实现偏好

- 先扩展设计系统，再做页面局部 workaround
- 能复用现有模式，就不要为单个场景发明新视觉语言
- 没有明确需求时，不改变现有功能语义和操作路径
- 新视觉角色优先进入共享 token 或共享组件，而不是页面私有实现
- 局部例外必须显式、可解释，而且尽量变薄

---

## 13. 给 Codex 与后续协作者的默认约束

在本文语境下，默认执行约束是：

- 不为单页需要新增一次性视觉规则
- 不把 Quiet Pro 当成“只能保守不能优化”的借口，而是当成可复用系统
- UI 问题修复时仍应遵守 [`issue-fix-boundary-guardrails.md`](./issue-fix-boundary-guardrails.md) 与 [`architecture.md`](./architecture.md) 的 owner 规则
- 如果某个方案明显偏离 Quiet Pro，应先停下确认，而不是直接落地
