## 3. 第一性原理

### 3.1 导出的本质

导出不是备份，也不是简单复制数据库列。导出的本质是：

> 在不改变 Patina 数据语义的前提下，把同一组活动记录转换为适合不同外部用途的文件表达。

因此必须先固定共同语义，再实现不同序列化方式。

### 3.2 用户真正需要控制的四件事

用户进行数据导出时，依次需要决定：

1. 导出哪段时间。
2. 文件准备拿去做什么，因此选择什么格式。
3. 文件包含哪些字段以及字段顺序。
4. 文件保存在哪里。

界面顺序必须与这个决策顺序一致。

### 3.3 共同逻辑数据与格式表达分离

- 32 个字段构成当前共同逻辑字段目录。
- 四种格式不得擅自改变字段的业务含义。
- 格式可以改变类型表达、空值表达、可读格式和文件结构。
- 相同时间范围和相同字段选择应基于同一批逻辑记录。
- Markdown 不是另一套数据源，而是同一逻辑数据的可读文档表达。

### 3.4 可信优先

- 导出不得修改源数据库。
- 导出失败不得留下看似成功的半成品。
- 空数据必须有明确且一致的处理结果。
- 当前活动、跨范围活动和网页活动必须沿用现有时间重叠口径。
- 现有 CSV、SQLite、Parquet 的兼容行为不得因新增 Markdown 静默回归。

## 6. 当前 32 个逻辑字段契约

### 6.1 字段数量来源

当前 `DATA_EXPORT_PROTOCOL_FIELDS` 与 Rust `ALL_EXPORT_FIELDS` 共同定义 32 个字段。数字 32 是现有协议事实，不是长期上限，也不是要求每次导出全部字段。

本任务必须保证：

- [x] 前端和 Rust 的字段集合完全一致。
- [x] 每个字段在中英文界面都有名称和说明。
- [x] 未知字段在 Rust 边界被拒绝。
- [x] 重复字段被去重，并保留第一次出现的位置。
- [x] 空字段选择在所有格式中被一致拒绝。

字段配置界面不再使用“默认／分析／高级”三组平铺所有语义。改为以下六组；协议字段名保持不变。

## 7. 四种格式的内容契约

## 7.1 共同记录口径

- [x] 应用会话和网页片段进入同一逻辑活动记录流。
- [x] `record_type` 区分 `session` 与 `web`。
- [x] 使用现有重叠范围规则，不改为“开始时间必须落在范围内”。
- [x] 当前未结束记录使用现有导出截止时间口径。
- [x] 分类名称、分类 ID 和分类颜色使用导出时的当前分类解析结果。
- [x] 四种格式使用同一字段验证与去重函数。
- [x] 四种格式返回同一逻辑记录数量。
- [x] 格式层不得单独过滤某类记录。

## 7.2 CSV 契约

- [x] 文件扩展名为 `.csv`。
- [x] 使用一行表头和一行一条活动记录的扁平结构。
- [x] 表头使用稳定协议字段名，不使用随语言变化的显示标签。
- [x] 列顺序严格等于用户字段顺序。
- [x] 时间字段继续使用现有稳定文本格式。
- [x] 数值字段保持机器可计算形式。
- [x] 空值输出为空单元格。
- [x] 按现有 Excel 安全规则处理公式注入风险。
- [x] 正确引用逗号、双引号、换行和长文本。
- [x] UTF-8 行为与现有导出兼容。

## 7.3 SQLite 契约

- [x] 文件扩展名默认为 `.sqlite`，保存对话框兼容 `.db`。
- [x] 输出为独立可打开的 SQLite 数据库。
- [x] 保持现有活动表结构和表名兼容，除非实施前测试证明必须调整。
- [x] 只创建用户选择的业务字段列。
- [x] 列顺序按用户选择顺序生成。
- [x] 整数、文本和空值使用真实 SQLite 类型语义。
- [x] 时间与时长字段保持既有机器可查询表达。
- [x] 重复导出到同一路径时替换旧输出，不追加历史残留。
- [x] 事务失败时不留下部分写入的正式目标文件。
- [x] 不把导出 SQLite 描述为 Patina 备份。

## 7.4 Parquet 契约

- [x] 文件扩展名为 `.parquet`。
- [x] 一条逻辑活动对应一行。
- [x] Schema 字段顺序严格等于用户字段顺序。
- [x] 字符串字段使用 UTF-8 字符串类型。
- [x] ID、时间戳和毫秒时长使用稳定整数类型。
- [x] `duration_minutes` 使用既有稳定数值类型与精度。
- [x] 空值使用 Parquet null，不使用空字符串冒充未知值。
- [x] 输出可由仓库现有 Arrow/Parquet 读取测试重新打开。
- [x] 字段增加不得改变未选择字段的输出。

## 7.5 Markdown 契约

- [x] 文件扩展名为 `.md`。
- [x] 使用 UTF-8。
- [x] 文档遵守 CommonMark 基础语法，并使用 GitHub Flavored Markdown 表格。
- [x] 文档标题固定表达为 Patina 活动记录，不使用用户数据充当一级标题。
- [x] 文档头部包含导出时间范围、导出时间、记录数量和总时长摘要。
- [x] 活动按本地日期升序或降序的规则必须明确并固定；推荐按日期升序、日内按开始时间升序。
- [x] 每个日期使用二级标题。
- [x] 每个日期下使用一张活动表格。
- [x] 表格列严格对应用户选择字段及顺序。
- [x] 用户可见表头使用当前界面语言的字段标签。
- [x] 协议字段语义不因标签语言变化。
- [x] 应用与网页专属字段不适用时显示 `—`。
- [x] 时间字段转换为适合阅读的本地时间文本。
- [x] `duration_ms` 与 `duration_minutes` 均保持各自字段语义；仅显示格式可读化，不允许把两列合并成一个字段。
- [x] URL 字段使用安全 Markdown 链接或经过转义的纯文本；实现前固定一种规则。
- [x] 空数据仍输出标题、元信息和“所选范围内没有活动记录”。
- [x] 记录数量仍按逻辑活动记录计算，不按日期表格数量计算。

## 8. Markdown 转义与边界规则

- [x] 转义表格单元格中的 `|`。
- [x] 处理反斜杠，避免形成错误转义。
- [x] 把 `\r\n`、`\n`、`\r` 规范为单个可读空格或明确的 `<br>`；推荐规范为空格。
- [x] 不允许单元格内容插入新的 Markdown 表格行。
- [x] 标题中的 `#`、反引号、方括号、括号和星号不得破坏文档结构。
- [x] URL 中的括号、空格和非 ASCII 字符必须产生可打开或至少不破坏文档的文本。
- [x] 不截断长标题、长 URL 或长应用名称。
- [x] 不把 HTML 当作默认逃生路径，除非仅用于稳定的换行表达且有测试。
- [x] 保留 Unicode、中文、emoji 和组合字符。
- [x] 防止活动内容伪造新的文档标题、表头或链接目标。
- [x] 特殊字符测试必须覆盖 `|`, `\\`, `` ` ``, `[]()`, `#`, 换行、emoji 和超长文本。

## 9. 各格式默认字段与独立偏好

### 9.1 偏好模型

前端本地偏好至少包含：

- [x] 上次使用的导出格式。
- [x] 上次使用的时间范围模式。
- [x] CSV 字段选择与顺序。
- [x] SQLite 字段选择与顺序。
- [x] Parquet 字段选择与顺序。
- [x] Markdown 字段选择与顺序。

建议的持久化形状：

```text
patina:export-format
patina:export-range-mode
patina:export-fields:csv
patina:export-fields:sqlite
patina:export-fields:parquet
patina:export-fields:markdown
```

- [x] 偏好继续使用 localStorage best-effort 语义。
- [x] localStorage 不可用时回退到各格式默认值，不阻塞导出。
- [x] 非法 JSON、未知字段、重复字段和空数组必须安全归一化。
- [x] 读取旧版本只有格式和范围偏好的状态时保持兼容。
- [x] 不保存具体自定义起止日期，避免下次打开默认使用过时范围。

### 9.2 默认字段原则

- [x] CSV 默认兼顾常用表格查看。
- [x] SQLite 默认适合本地查询和较完整归档，但不得宣称是备份。
- [x] Parquet 默认适合分析字段和稳定标识。
- [x] Markdown 默认适合直接阅读，减少技术标识字段。
- [x] 用户切换格式时加载该格式上次保存的字段，而不是覆盖成当前格式字段。
- [x] 每种格式均提供“恢复该格式默认字段”动作。
- [x] 默认字段数组必须是 32 字段目录的合法子集。
