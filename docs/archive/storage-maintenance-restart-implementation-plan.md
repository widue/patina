## 2. 已确认的产品决策

### 2.1 覆盖范围

本项统一覆盖以下五类操作：

1. 更改数据目录。
2. 恢复默认数据目录。
3. 更改 WebView 缓存目录。
4. 恢复默认 WebView 缓存目录。
5. 清理 WebView 缓存。

### 2.2 用户可见交互

所有目标操作的最终确认弹窗只提供：

- `取消`
- `重启并应用`

以下行为全部等价于取消：

- 点击“取消”。
- 点击允许关闭的弹窗遮罩。
- 点击弹窗关闭入口。
- 按 `Esc`。

取消必须满足：

- 不写入 pending。
- 不改变当前目录锚点或缓存维护状态。
- 不退出或重启 Patina。
- 不显示“已安排下次启动执行”之类成功提示。

### 2.3 pending 的产品边界

用户界面不再暴露“稍后重启”“下次启动应用”“待执行迁移”或长期 pending 管理入口。

技术实现仍允许使用短生命周期 pending，目的只有两个：

1. 把已确认操作从当前进程安全交接给即将启动的新进程。
2. 自动重启拉起失败时，为本次已确认操作提供一次故障恢复依据。

pending 是内部事务记录，不是用户任务列表。

### 2.4 兼容决策

- 旧格式 pending 不兼容。
- 不迁移旧格式字段或语义。
- 不执行旧格式 pending 中的目录或缓存操作。
- 检测到旧格式 pending 时，安全清理并记录可解释的维护诊断。
- 新流程刚刚写入的新格式 pending 不属于旧兼容范围，必须支持故障恢复。

### 2.5 明确非目标

- 不允许当前进程热迁移正在使用的 SQLite 或 WebView2 资源。
- 不为“稍后重启”保留入口、状态徽标或管理弹窗。
- 不建立一套与现有 `run_pending_storage_migration` 并行的迁移机制。
- 不借本项重构全部启动生命周期或 tracking runtime。
- 不改变秒表、番茄钟等工具在正常退出和重新启动后的既有语义。
- 不顺带改变备份格式、数据库 schema 或更新安装流程。
- 不引入页面私有颜色、按钮原型或弹窗视觉语言。

## 3. 第一性原理

### 3.1 资源所有权决定操作时机

当前进程运行期间：

- SQLite 数据库可能持有连接、WAL 文件和未完成事务。
- WebView2 运行时持有 `EBWebView` 内的文件与句柄。
- tracking、工具和桥接运行时可能仍在产生写入。

只要资源仍被当前进程拥有，强行移动或删除就无法同时保证完整性、可预测性和可恢复性。因此，目录迁移和 WebView 缓存维护必须在旧进程释放资源之后、新窗口和常驻运行时建立资源所有权之前执行。

由此得到第一个不变量：

> 目标操作只能在新进程的启动前维护窗口中执行，不能在当前 UI 进程中热应用。

### 3.2 跨进程意图必须持久化

内存状态会随旧进程退出而消失。要让新进程知道用户确认了什么，必须存在一个落盘交接记录。

由此得到第二个不变量：

> 只有 pending 已经原子写入且能够回读确认后，旧进程才有资格退出。

### 3.3 用户确认必须对应单一结果

用户选择“重启并应用”，表达的是一个整体意图，而不是“先安排，之后由我记住并重启”。界面不应把内部跨进程事务拆成用户需要管理的两步工作。

由此得到第三个不变量：

> 正常路径中，确认、提交、退出、重新拉起和应用属于一个产品操作。

### 3.4 取消必须是真正的零副作用

如果预览阶段已经写 pending，用户随后取消就需要补偿或回滚；一旦补偿失败，取消便不再可信。因此预览必须保持只读，提交只能发生在“重启并应用”之后。

由此得到第四个不变量：

> 预览和取消不落盘；确认后才进入不可逆的提交阶段。

### 3.5 安全优先于表面原子性

跨进程操作无法与操作系统进程拉起形成真正的单个 ACID 事务。可以保证的是：

- pending 写入采用原子文件替换。
- 未确认落盘时不退出。
- 已确认落盘后即使拉起失败，也不丢失本次请求。
- 执行失败时保留可信源数据并记录结果。
- 执行成功后清除 pending，避免重复执行。

因此本项追求的是“可恢复的事务边界”，而不是假装进程重启可以完全原子化。

### 3.6 幂等是重启安全的基础

UI 禁用按钮只能降低重复点击概率，不能抵御 IPC 重放、竞态、异常恢复或外部重复调用。Rust owner 必须拒绝或安全合并重复请求，并保证同一个事务 ID 不会被重复应用。

由此得到第五个不变量：

> 前端负责清晰反馈，Rust 负责最终幂等和状态权威。

## 5. 目标状态机与不变量

### 5.1 状态机

```text
Idle
  └─ 用户发起操作 → Previewing

Previewing
  ├─ 预览失败 → Idle + 显示错误
  └─ 预览成功 → AwaitingConfirmation

AwaitingConfirmation
  ├─ 取消/关闭/Esc → Idle（零副作用）
  └─ 重启并应用 → Committing

Committing
  ├─ 校验或 pending 写入失败 → AwaitingConfirmation + 显示错误
  └─ 写入并回读成功 → RestartRequested

RestartRequested
  ├─ 旧进程退出并成功拉起 → StartupMaintenance
  └─ 拉起失败 → PendingForRecovery

PendingForRecovery
  └─ 用户下次手动启动 → StartupMaintenance

StartupMaintenance
  ├─ 执行成功 → 清除 pending + 记录成功 → Running
  └─ 执行失败 → 保留可信源 + 清除或终结本事务 + 记录失败 → Running
```

### 5.2 必须长期成立的不变量

- [x] `AwaitingConfirmation` 之前没有写侧副作用。
- [x] pending 写入失败时不得退出。
- [x] pending 回读内容必须与本次事务 ID、操作种类和目标一致。
- [x] `RestartRequested` 只能由一个原子状态门进入一次。
- [x] 新窗口创建前完成 pending 的读取、格式判断和维护尝试。
- [x] 旧格式 pending 永不触发迁移或删除动作。
- [x] 数据迁移失败时，源数据库和必要数据文件仍然存在。
- [x] 成功事务不会在后续启动中再次执行。
- [x] 失败事务不会形成无限启动重试循环。
- [x] UI 不把内部 pending 表述为用户需要稍后完成的工作。

## 阶段 1：定义新事务契约与 pending 格式

目标：先定义权威状态，再连接重启行为。

- [x] 1.1 定义统一的操作种类。
  - 至少能区分数据目录迁移、缓存目录迁移、恢复默认数据、恢复默认缓存和缓存清理。
  - 如果目录迁移继续使用“最终目标快照”合并模型，应明确操作种类是审计信息还是执行依据。
  - 完成判据：Rust 类型无法表达未知字符串操作而不经过显式解析。

- [x] 1.2 引入新格式标识。
  - 使用新的明确格式常量，例如 `patina.storage-restart-operation.v2`；最终命名以现有命名体系为准。
  - 不复用旧格式常量假装兼容。
  - 完成判据：新旧格式可被确定地区分。

- [x] 1.3 定义新 pending 最小字段。
  - 建议包含：格式、事务 ID、创建时间、操作种类、源数据根、目标数据根、目标 WebView 根、缓存清理标志和状态。
  - 不存储可从稳定路径重新计算的页面文案。
  - 完成判据：新进程仅凭 pending 和稳定锚点即可确定要做什么。

- [x] 1.4 定义状态枚举。
  - 最少区分 `prepared` 与执行终态；避免继续依赖任意字符串 `pending-restart`。
  - 如果文件只在 prepared 时存在，应让“文件存在”与“准备完成”语义一致。
  - 完成判据：非法状态被拒绝，不触发文件操作。

- [x] 1.5 保持原子写入。
  - 复用 `storage_anchor` 现有原子 JSON 写入能力。
  - 写入后立即回读并比较事务 ID、格式和关键目标字段。
  - 回读失败或不一致时删除本次未确认状态并返回错误；不得发起重启。
  - 完成判据：测试覆盖完整写入、损坏 JSON、字段不一致和替换失败。

- [x] 1.6 明确同一时刻只允许一个重启事务。
  - 没有 pending 时允许创建。
  - 已有同一事务的重复请求应返回幂等成功或“正在准备重启”。
  - 已有不同新格式事务时应拒绝，不能静默覆盖。
  - 完成判据：并发测试证明最终只有一个权威 pending。

## 阶段 2：旧 pending 明确退出

目标：落实“不兼容旧 pending”，同时避免旧文件触发危险操作。

- [x] 2.1 在任何迁移或删除前先解析格式。
  - 格式缺失、旧格式或未知格式都不得进入执行函数。
  - 不根据旧字段猜测用户意图。
  - 完成判据：旧 fixture 不会触碰源目录、目标目录或缓存目录。

- [x] 2.2 定义旧 pending 清理顺序。
  - 先读取并判定为不支持。
  - 再删除旧 pending 文件。
  - 最后写入维护诊断，说明旧待执行操作已被丢弃且未执行。
  - 诊断写入失败不得反过来执行旧操作。
  - 完成判据：下次启动不会重复处理同一个旧 pending。

- [x] 2.3 清理旧的用户可见 pending 能力。
  - 移除 Settings pending 徽标、状态弹窗和“取消待执行迁移”入口。
  - 移除只为长期 pending UI 服务的前端 DTO 字段；若诊断仍需字段，应改成明确的维护结果模型。
  - 评估并删除不再可达的 `cmd_cancel_pending_storage_migration`，不得保留死入口。
  - 完成判据：生产 UI 和 IPC 中不存在让用户管理旧 pending 的入口。

- [x] 2.4 为退出行为增加测试。
  - 旧格式文件被删除。
  - 源、目标和缓存内容均未变化。
  - 维护错误/状态可从存储快照读到。
  - 未知未来格式同样 fail closed。
  - 完成判据：测试证明“不兼容”不等于“无保护地忽略”。

## 阶段 3：实现 Rust 受控重启事务

目标：让“提交操作并重启”成为单一 Rust 语义，而不是两个松散前端调用。

- [x] 3.1 选择并验证 Tauri v2 重启机制。
  - 优先使用 Tauri 本身提供的应用重启能力；若当前依赖能力不足，再评估官方 process 插件。
  - 不使用前端自行拼接可执行文件路径或 `Command` 启动程序。
  - 验证 Windows release 构建下的行为，不只验证 `tauri dev`。
  - 完成判据：单实例、当前可执行文件路径和参数传递行为均有证据。

- [x] 3.2 增加应用级重启协调状态。
  - 使用原子布尔值、互斥状态或等价机制确保只接受一次重启请求。
  - 状态 owner 位于 `app/*`，不放在 command 或前端。
  - 定义“准备失败后是否释放重启锁”；建议 pending 未提交时释放，已提交后保持终态。
  - 完成判据：两个并发 IPC 请求最多一个进入提交路径。

- [x] 3.3 定义统一的提交函数。
  - 输入为经过预览的具体请求，而不是让前端传入任意 pending JSON。
  - 重新解析当前路径并再次执行权威校验，不能信任旧预览。
  - 创建新事务 ID。
  - 原子写 pending 并回读确认。
  - 标记应用退出为受控退出。
  - 发起重启。
  - 完成判据：任何重启调用都发生在 pending 回读成功之后。

- [x] 3.4 保持 command 薄。
  - 可提供五个语义明确的 command，或一个带严格 enum DTO 的 command。
  - command 只做参数映射和调用应用/data owner。
  - 不在 command 中直接读写文件、创建备份或处理路径包含关系。
  - 完成判据：`check:rust-boundaries` 不需新增例外。

- [x] 3.5 处理提交失败。
  - 路径在预览后失效、目标被占用、原子写入失败、回读失败时返回结构化错误。
  - 不调用退出或重启。
  - 清理只属于本次、且尚未确认的临时状态。
  - 完成判据：应用保持运行，用户可修改选择后重试。

- [x] 3.6 处理重启拉起失败。
  - 已确认 pending 不回滚，因为旧进程可能已进入退出阶段。
  - 允许用户下次手动启动时由新格式恢复流程执行。
  - 在可行范围记录诊断；不得通过循环拉起制造重启风暴。
  - 完成判据：故障注入后 pending 保留且只执行一次。

- [x] 3.7 核验正常退出与记录可信度。
  - 受控重启必须走正常退出语义或等价的可靠封口路径。
  - 验证活动 session 不会异常延长到整个停机期间。
  - 验证重启后不会复制上一 session 的活动区间。
  - 完成判据：tracking replay 或针对性测试覆盖重启边界。

- [x] 3.8 处理 updater 竞争。
  - 明确存储维护重启与更新安装重启不能同时进入。
  - 如果 updater 正在安装或已持有重启权，存储操作应被拒绝并给出可理解反馈。
  - 不在本项合并两类 pending 格式。
  - 完成判据：测试或状态机证明不存在两个重启 owner 同时拉起应用。

## 阶段 4：统一启动前维护

目标：新进程安全消费新事务，并让失败可解释、不可循环。

- [x] 4.1 保持执行位置早于资源占用。
  - 新 pending 的读取和执行继续位于 `runtime::setup`、主窗口创建及 SQLite 常驻使用之前。
  - 不把维护推迟到 React 页面加载后。
  - 完成判据：代码顺序和集成测试共同证明执行窗口正确。

- [x] 4.2 统一目录迁移与缓存清理计划。
  - 一个事务可以准确表达本次需要的数据目录目标、缓存目录目标和是否清理缓存。
  - 如果目标操作只有缓存清理，不应生成伪数据迁移。
  - 如果目标操作只改变数据目录，不应误删当前 WebView 缓存。
  - 完成判据：五类操作矩阵中的非目标资源保持不变。

- [x] 4.3 数据目录迁移继续执行预迁移备份。
  - 在破坏性清理旧数据前完成可验证备份。
  - 备份失败则终止迁移，不改变活动锚点，不删除源数据。
  - 完成判据：备份失败测试证明源路径仍是权威路径。

- [x] 4.4 定义 pending 的终结规则。
  - 成功：完成锚点切换和必要清理后删除 pending，再记录成功。
  - 可恢复失败：保留源数据，但不得每次启动无限重试同一危险步骤；将事务转为终结失败并移除可执行 pending。
  - 进程在执行中崩溃：下次启动通过事务 ID 和已完成事实进行幂等判断，不能盲目重复破坏性步骤。
  - 完成判据：测试覆盖每个关键步骤前后崩溃或错误返回。

- [x] 4.5 定义应用启动保障。
  - 维护失败默认不阻止 Patina 使用仍可信的路径启动。
  - 如果无法解析出可信数据路径，应 fail safe，并给出明确诊断；不得创建空数据库掩盖原数据。
  - 完成判据：迁移失败场景不会让唯一数据看似“消失”。

- [x] 4.6 输出可消费的维护结果。
  - 至少区分成功、失败、旧 pending 已丢弃。
  - 包含用户可理解的摘要和供诊断的稳定错误码/阶段。
  - 不把原始系统错误或敏感完整路径无节制暴露到 UI。
  - 完成判据：Settings 能根据结构化结果给出确定反馈。

## 9. 故障注入与恢复验证

### 9.1 故障点清单

实现者应为以下边界建立可测试的注入点或等价测试替身：

- pending 临时文件创建失败。
- pending 原子替换失败。
- pending 回读失败或内容不一致。
- 预迁移备份失败。
- 目标目录创建失败。
- 数据复制中途失败。
- 锚点切换失败。
- 旧目录清理失败。
- WebView 缓存删除失败。
- 发起重启失败。
- 新进程在维护中途退出。

### 9.2 每个故障点必须回答

- [x] 旧进程是否仍然运行。
- [x] pending 是否存在、是否可执行、是否会无限重试。
- [x] 当前权威数据路径是什么。
- [x] 源数据是否完整。
- [x] 下次启动会做什么。
- [x] 用户能看到什么结果。
- [x] 日志或维护状态是否足以诊断。

### 9.3 禁止的恢复方式

- 不允许在信息不足时猜测哪一份数据库更新。
- 不允许为“保证启动”静默创建空数据库覆盖错误表象。
- 不允许迁移失败后先删除源数据再尝试补救。
- 不允许未知 pending 格式进入破坏性路径。
- 不允许每次启动无上限重试同一失败事务。

## 10. 回归风险及防护

| 风险 | 触发原因 | 防护 |
| --- | --- | --- |
| 操作丢失 | pending 未确认落盘就退出 | 写入后回读，再允许重启 |
| 重复迁移 | 多次点击或崩溃恢复重复执行 | Rust 重启锁、事务 ID、幂等检查 |
| 数据看似丢失 | 迁移失败后打开空目标数据库 | 失败保留源锚点，禁止静默空库 |
| 无限启动循环 | 失败 pending 每次启动自动重试 | 明确失败终结状态和重试上限 |
| 多实例 | 手工启动与自动拉起竞争 | 单实例插件 + 应用级重启状态 |
| session 异常延长 | 重启绕过正常退出语义 | 复用退出路径并验证 tracking 边界 |
| UI 卡死 | 重启命令中断 IPC 返回 | 不依赖成功返回，处理 Promise 终止 |
| 旧 pending 被误执行 | 复用旧解析路径 | 格式先行、fail closed、无副作用清理 |
| updater 冲突 | 两个 owner 同时要求重启 | 显式互斥和用户反馈 |
| 架构回流 | 为求方便把时序堆进 command/app 壳 | owner-first、边界检查、局部模块化 |

## 12. 建议提交拆分

具体提交以最终 diff 为准，但默认建议按可独立评审的行为拆分：

1. `refactor(storage): define restart operation transaction`
   - 新 pending 契约、格式判断、原子提交和旧格式退出测试。
2. `feat(storage): restart and apply maintenance operations`
   - 应用级重启协调、启动前消费、失败恢复和 Rust 测试。
3. `feat(settings): require restart to apply storage changes`

4. `test(storage): cover Windows restart maintenance flows`
